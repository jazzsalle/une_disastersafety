## 1. 추진 배경 및 목적

### 1.1 배경 (사업 컨텍스트)
- 본 POC는 NIA 발주 「재난안전 AI데이터 구축」 사업(2026.05.26~2027.01.21, 240일, 티쓰리큐 컨소시엄)의 일부다. 사업은 Event-Centric 통합 식별체계·OCR/NLP 문서 구조화·GeoAI·RAG·Ontology/KG 기반으로 재난 데이터를 사건 단위로 결합한다.
- **컨소시엄 내 유엔이(UNE) 담당 = AI Agent 실증 서비스·시각화**(포털 화면, 구축현황 대시보드, 데이터 조회·검색·시각화, 실증·행사 시연). 본 저장소는 이 실증 서비스 화면의 선행 POC다.
- RAG 엔진·AI Agent/MCP·VectorDB API는 티쓰리큐가 제공 예정 → POC의 자체 검색·응답 구현은 **T3Q RAG API 연계 전까지의 대체 구현(mock)**이며, API 계약을 연계 규격에 맞춰 설계한다.
- 지자체 재난 대응의 핵심 참조 문서인 **자연재해저감종합계획**과 **하천기본계획**은 DAR-008 문서 구조화 대상(약 3만 건)으로 명시되어 있고, 수백~수천 페이지 보고서라 상황 발생 시 필요한 정보(위험지구, 계획홍수량, 저감대책)를 즉시 찾기 어렵다.

### 1.2 목적
재난 상황 입력 → 유사사례·관련 문서 검색(RAG) → 근거 기반 응답 → **재난메타+데이터메타 기준의 지도·시각화 표출**까지의 전 과정을 시범 구현하여,
1. 계획 문서(비정형 PDF)를 재난메타 체계로 구조화·검색 가능함을 실증하고,
2. 판단기준(특보·등급)과 결합한 상황 표출 화면의 UX 방향을 제시하며,
3. 사업수행계획서의 시범서비스 모듈 5종(입력 패널·지도 뷰어·Top-K 목록·상세조회·정합성)의 구현 방향을 검증한다.

## 3. 사용자 및 시나리오

### 3.1 대상 사용자
- **1차**: 지자체·중앙부처 재난상황실 담당자 (상황 판단·보고 지원)
- **2차**: 방재 계획 담당 공무원, 유관기관 분석가

### 3.2 핵심 사용자 시나리오
**S1. 호우 상황 — 위험지구 조회 (의왕)**
> 의왕시에 호우경보(3h 90mm 초과 예상) 발효. 담당자가 재난상황 패널에 [호우 / 의왕시 / 경보]를 입력하고 "침수 위험이 큰 지구와 저감대책은?"을 질의 → 저감종합계획 내 내수재해·하천재해 위험지구 목록, 지구별 위험요인·저감대책·사업비가 카드/표로 표출되고, 근거 페이지가 인용된다.

**S2. 홍수 상황 — 하천 기준 조회 (남원/요천)**
> 요천 수위 상승. [홍수 / 남원시 / 주의보]를 입력하고 "요천의 계획홍수량과 주의보 기준은?"을 질의 → 하천기본계획의 지점별 계획홍수량 표와, 판단기준(계획홍수량 50%/70% = 주의보/경보 수위 개념)이 함께 표출된다.

**S3. 산사태 상황 — 사면재해 대책 조회 (구미)**
> 토양함수지수 90%(예비경보) 도달. [산사태 / 구미시 / 예비경보] 입력, "사면재해 위험지구 현황과 등급별 대응은?" 질의 → 저감종합계획의 사면재해 위험지구, 등급(가/나/다), 판단기준 3단계(80/90/100%)가 게이지·배지로 표출된다.

## 4. 핵심 기능 (1차 POC)

사업수행계획서의 시범서비스 모듈 5종(①입력 패널 ②지도 뷰어 ③Top-K 목록 ④상세조회 ⑤정합성)을 기준으로 기능을 정의한다.

| # | 기능 (모듈) | 설명 |
|---|---|---|
| F1 | 재난상황 입력 패널 (①Input Panel) | 재난유형(5종)·지자체·경보단계·발생시간·기상 조건·피해/통제 키워드(예: 시간당 50mm, 침수, 지하차도)를 구조화 입력 → RAG 질의 JSON 생성(Query Builder). 재난메타 사건(EVT) 골격 대응 |
| F2 | 검색·유사사례 (RAG Retrieval) | 저감계획·하천기본계획 코퍼스에서 상황 조건(유형·지자체 필터)+질의로 관련 청크·유사 사례 검색. T3Q RAG API 연계 규격의 mock 구현 |
| F3 | 근거 기반 응답 + Top-K 목록 (③Top-K Results) | 검색 결과를 유사도 점수와 함께 카드 목록으로 제시하고, LLM이 근거 기반 응답 생성(근거 없는 답변 금지). API 미설정 시 결정적 mock 폴백 |
| F4 | 판단기준 시각화 | 선택 유형의 판단조건·등급(판단기준_5종)을 기준표·게이지·단계 배지로 표출 |
| F5 | 상세조회 (④Case Detail) | 위험지구(지구명·유형·등급·위험요인·저감대책·사업비·우선순위)·하천(제원·계획홍수량·홍수위) 정형 카드 + 근거 문서 위치 |
| F6 | 정합성·근거 추적 (⑤Verification) | 모든 응답에 출처(문서명·장·페이지) Citation + 유사도 요인별 기여 표시 — "왜 이 결과인지" 검증 UI |
| F7 | 지도 통합 뷰어 (②Map Viewer) | **VWorld 2D 지도(국토부 오픈플랫폼) 기반**으로 위험지구·하천 구간·유사사례 위치를 표출. L1 좌표/시설(마커)·L2 구간/면(폴리곤)·L3 행정구역·L4 광역/불명확(목록 폴백)의 공간표현 전략 적용 |

### 제외 (1차 POC 범위 외)
- 실시간 관측 연계(AWS·수위관측 API), 사용자 인증·RBAC(공개용 화면 기준 구현, 내부용 정보는 정합성 패널 토글로 표시 수준), HWP 파싱, 부산/인제/영천 확장, 상황일지·중대본보고서·위성영상 코퍼스, T3Q RAG API 실연계(계약만 정합), Knowledge Graph·VectorDB(고도화 시 T3Q 스택 연계)

## 6. 성공 기준 (1차 POC)
1. 3개 시나리오(S1~S3)가 데모 가능: 입력→검색→응답→시각화 전 과정 동작
2. 코퍼스: PDF 4종(구미·의왕 저감계획, 요천·구미천 하천기본계획) 청킹·메타 태깅 완료
3. **근거 제시율 100%** — 모든 AI 응답에 출처(문서명·장·페이지) Citation, 근거 없는 답변 금지(사업수행계획서 XAI 원칙)
4. UI가 UNE Design System 토큰·컴포넌트와 SpoqaHanSansNeo 글꼴만으로 구성
5. 판단기준 5종이 화면에서 조회 가능
6. 시범서비스 모듈 5종(입력 패널·지도 뷰어·Top-K 목록·상세조회·정합성)이 화면에 모두 구현
7. 서비스 검증 체크리스트(사업수행계획서 8단계: 질의 구조화→API 응답→결과 수신→위치 메타→지도 표출→근거 확인→응답 정합성→로그) 통과
