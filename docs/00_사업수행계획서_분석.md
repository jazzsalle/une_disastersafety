## 4. 시범서비스 명세 (4.2(7) AI Agent/MCP 기반 시범서비스 수행방안) — ★화면설계 직접 근거

서비스 모듈 5종과 데이터 흐름:

| 모듈 | 주요 기능 | 구현 요점 |
|---|---|---|
| ① 재난상황 입력 패널 (Input Panel) | 재난유형·지역·시간·기상 조건·피해/통제 키워드 구조화 입력 (예: "시간당 50mm 강우, 침수, 하수 역류, 차량고립, 지하차도, 반지하") | 복잡한 RAG 질의를 입력 패널로 전환 → 구조화 JSON 질의 생성(Query Builder) |
| ② 지도 통합 뷰어 (Map Viewer) | 유사사례 위치·취약지역 GIS·영상 촬영 위치·위험등급 시각화 | **L1~L4 공간표현 전략**: L1 좌표/시설(마커) · L2 구간/면(폴리곤) · L3 행정구역 · L4 광역/위치불명확(목록) |
| ③ 유사사례 목록 (Top-K Results) | 매칭 사건 리스트·유사도 점수·대응 내역 요약 | 결과 카드, 정렬, 필터, 상세 팝업 |
| ④ 상세정보 조회 (Case Detail) | 피해·대응·기상·영상 메타·근거 문서 조회 | 공개등급에 따른 노출 범위·출처 링크 |
| ⑤ 정합성 (Verification) | 유사도 그래프·요인별 기여도·Citation(출처·근거) | "왜 유사한지"를 정성·정량으로 검증하는 UI |

- 처리 흐름: 질의 이해 → 질의 구조화(JSON) → AI Agent/RAG API 도구 호출(유사사례 검색·공간분석·사건관계 탐색·상세조회) → 응답 수신(Top-K·유사도·근거 Passage·지도 레이어 GeoJSON) → 결과 통합 → 시각화·로그
- **공개용 화면 / 내부 검증용 화면 분리** + 개인정보 필터링 + 역할별(관리자/행안부·책임기관/연구자·민간/대국민) RBAC 노출 차등
- 서비스 검증 체크리스트 8단계: 질의 구조화 → API 호출·응답 → 검색결과 수신 → 위치 메타데이터 → 지도 시각화 → 근거 Passage·출처 → AI 응답 정합성(환각) → 로그
- 교육: 사용자/관리자/운용자 매뉴얼 제작 포함
- TER-004: **RAG 구축 후 1년간 재난 유사사례 검색 시범서비스 실시**

## 5. Event-Centric 데이터 체계 (DAR-001) — API·스키마 정렬 기준

- **Event Master ID 형식**: `[재난유형코드]-[발생일시]-[지역코드]-[순번]` 예: `FLOOD-20250718-11680-0001`
- 상태값: provisional → confirmed / merged / split / deprecated
- Event Master 속성: event_id·parent_event_id / disaster_type·subtype / occurred_at·ended_at / admin_code·geometry·bbox·crs / casualties·damage_summary / source_count·confidence_score / document_ids·warning_ids·risk_area_ids·graph_node_id·vector_index_id
- **동일 사건 매칭**: Event Match Score = 유형·시간·공간·피해·경보·기관·문맥·멀티모달 유사도 가중합 (검색 유사도 산정에도 동일 적용)
- 추적성 식별자 체계: `source_asset_id`(원본+SHA-256) → `evidence_unit_id`(페이지·문단·셀) → `passage_id`(RAG 근거) → `event_id` → `rag_answer_id`(응답). **근거 없는 답변 금지(XAI)**

## 6. 문서 구조화 (DAR-008) — 저감계획·하천기본계획 정형화 요구

- 대상: **자연재해저감종합계획·소하천정비종합계획·하천기본계획 PDF/HWP 약 3만 건** (전체 56만여 파일 중 우선 확보분). OCR 오류·표 구조 손실·좌표계 혼재·결측(-9999) 리스크 명시
- 정형화 항목: ①위험지역 지식 — 위험지역명·위험원인·발생조건·**위험조건(임계값: 수치+단위+비교연산+시점+대상)**·출처근거(문서·페이지·표·그림) ②재해예방사업 — 사업명·사업상태(구축완료/구축중/구축필요 의견)·조치계획·총사업비·연차별 계획·우선순위
- 계획/사례 구분: 계획 시점의 위험 예측정보 vs 실제 발생 사건을 구분해 Event ID로 1:N 연결
- → 본 POC의 `districts.json`(위험지구)·`rivers.json`(계획홍수량)이 DAR-008 정형화 항목의 부분집합임을 확인. 필드에 위험조건(임계값 구조체)·사업상태를 보강한다.

## 7. RAG 검증 체계 (TER-004) — POC 품질 기준

- 스택(오픈소스 지정): LangChain/LlamaIndex, Milvus/FAISS, OpenSearch(Hybrid), Neo4j(KG), PostGIS(Geo), Llama3(생성), BGE-M3/E5/SBERT(임베딩), Airflow(ETL)
- 검색 평가: Top-K 적중률, MRR, NDCG, **Geo Relevance**
- 응답 평가: **Groundedness, Faithfulness, Source Traceability, 근거 제시율** — 근거 Passage·요인 비교표 필수 제시
- 환각 점검: 응답 문장 단위 근거 Passage 대조 → 프롬프트·인덱스 보완 후 재검증
- 품질 기준(RFP): 사건 누락률 ≤1%, 주요 항목(일자·유형·위치) 정확도 ≥99% (DAR-003) / 내부 목표: OCR 신뢰도 ≥0.96, Passage 정합성 ≥98%, KG 연결율 ≥99%

## 8. 일정·마일스톤 (계약 240일)

M1 착수(승인: 사업수행계획서·WBS) → M2 요구분석(RTM) → M3 설계(데이터 표준·Event ID) → M4 1차 데이터 구축 → M5 전체 데이터셋(DAR-002~013) → M6 **RAG·GeoAI 검증 완료(시범서비스 검증 시점)** → M7 통합테스트 → M8 완료·검수. 유엔이 시범서비스는 9월~12월 RAG 검증 구간과 맞물림.

## 9. POC 기획·설계 반영 결론 (변경 요약)

1. **재난유형 라벨 조정**: 풍수해 → "수해/침수"로 표기(내부 T10106·INUNDATION 병기). UI 5종 = 호우·홍수·수해/침수·태풍·산사태
2. **화면 구성을 시범서비스 모듈 5종에 정렬**: 기존 3패널(입력·질의응답·시각화)에 ②지도 통합 뷰어(L1~L4 표현)와 ⑤정합성(요인 기여도·Citation) 추가, 질의응답을 ③유사사례 Top-K 목록+④상세조회 중심으로 재편
3. **API 계약을 T3Q RAG API 연계 규격으로 설계**: 구조화 질의 JSON(입력 패널→Query Builder) / 응답(Top-K 사례·유사도 점수·근거 Passage·GeoJSON 레이어·상세). POC 백엔드는 이 계약의 mock 서버 겸 자체 검색 구현
4. **식별자 체계 채택**: source_asset_id → evidence_unit_id → passage_id → event_id → rag_answer_id. Event ID 형식 `INUNDATION-YYYYMMDD-지역코드-순번`
5. **품질·검증 지표 반영**: evaluation_criteria에 근거 제시율 100%(근거 없는 답변 금지)·서비스 검증 체크리스트 8단계 반영
6. **공개용/내부용 화면 구분**: POC는 공개용 화면 기준으로 구현하되 내부용(검증 정보 확대) 토글을 정합성 패널에 반영
7. **districts/rivers 스키마 보강**: DAR-008 정형 항목(위험조건 임계값 구조체, 재해예방사업 상태) 반영
